# 日中漫画翻译 — 方式A vs 方式B 性能实测报告

**日期**: 2026-06-12  
**测试集**: 158 页漫画图片 (`test/e2e-materials/`)  
**硬件**: Apple M4 Pro / 64GB 统一内存 / macOS  
**配置**: `use_gpu_limited` (Detection/OCR/Inpainting → MPS, 翻译按方式选择)  
**Ollama 服务**: 192.168.1.15:11434, sakura-14b-qwen2.5-v1.0:latest  

---

## 1. 测试方法

分别以两种翻译方式运行完整翻译流水线（Detection → OCR → Translation → Inpainting → Render），各处理 158 张图片。

- **方式A (Ollama HTTP)**: 通过 OpenAI 兼容 API 调用远程 Ollama 服务
- **方式B (本地 GGUF)**: 通过 `llama-cpp-python` 直连 MPS 运行本地 GGUF 量化模型

测试脚本: `test/e2e_ollama.py` / `test/e2e_gguf.py`

## 6. 与历史测试对比

| 测试 | 日期 | 素材 | 方式B (GGUF) | 方式A (Ollama) | B vs A |
|------|------|------|-------------|---------------|--------|
| 157页 (第13话) | 2026-06-10 | 单话漫画 | 988.8s / 6.3s/页 | 1061.5s / 6.8s/页 | B 快 6.8% |

趋势分析：
- 早期测试中方式B 有可测量的速度优势（+4%~7%）
- 最新 158 页测试中两种方式差异缩小到 <1%，进入测量误差范围
- 可能原因：网络环境改善（Ollama 响应更快）、图片素材特征差异、系统负载波动

## 7. 综合结论

| 维度 | 结论 | 说明 |
|------|------|------|
| **速度** | **持平** | 差异 <1%，两种方式在实际使用中速度基本一致 |
| **稳定性** | **持平** | 双方均 100% 成功率，无崩溃/超时/截断 |
| **质量** | **持平** | 共用相同模型和 Prompt，质量无差异 |
| **首次启动** | 方式A 更优 | GGUF 需加载 ~10s，Ollama 随系统启动 |
| **部署复杂度** | 方式A 更优 | 无需额外安装 llama-cpp-python |
| **独立性** | 方式B 更优 | 不依赖 Ollama 服务，离线可用 |

### 推荐策略

- **日常使用**: 两种方式均可，速度差异可忽略
- **离线场景**: 使用方式B（GGUF），不依赖网络和 Ollama 服务
- **快速部署**: 使用方式A（Ollama），无需额外依赖
- **切换方法**: 设置/取消 `SAKURA_GGUF_PATH` 环境变量即可一键切换

### 修复前性能瓶颈分析

修复前 Qwen3 模型每次翻译请求会生成 200-300 个思考 token：
- 思考 token 生成：约 10-12s（占 API 总耗时的 80%+）
- 实际翻译 token 生成：约 2-3s（仅 2-3 个 token）
- 总 API 调用：15-27s/次

根因：`/v1/chat/completions`（OpenAI 兼容端点）不支持 `think` 或 `enable_thinking` 参数，模型始终以思考模式运行。

### 修复方案

将 `Qwen3KoZhTranslator._request_translation` 从 OpenAI 兼容端点改为 Ollama 原生 `/api/chat` 端点，直接传 `"think": false`：

```python
payload = {
    'model': self.model,
    'messages': messages,
    'stream': False,
    'think': False,  # 禁用思考模式
    'options': {
        'temperature': self.temperature,
        'top_p': self.top_p,
        'num_predict': self._MAX_TOKENS // 2,
    },
}
async with aiohttp.ClientSession() as session:
    async with session.post(self._ollama_chat_url, json=payload) as resp:
        data = await resp.json()
```

## 优化措施

### 1. 禁用 thinking 模式（关键优化）

- **效果**：API 调用从 15-27s 降至 0.6-2.9s，约 10x 提速
- **实现**：使用 Ollama 原生 `/api/chat` 端点 + `"think": false`
- **质量影响**：无明显影响（漫画翻译为确定性任务，不需要推理链）
- **局限性**：`/v1/chat/completions`（OpenAI 兼容端点）不支持此参数，必须使用原生端点

### 2. 温度参数优化

- **temperature = 0.3**：降低随机性，提高翻译一致性
- **top_p = 0.3**：限制采样范围，减少翻译偏差

### 3. 模型单次加载

- 模型加载一次（约 0.7s），整个批次复用
- 避免逐页重复加载开销

### 4. 进度续传

- 每页翻译完成后立即写入进度文件
- 中断后重启自动跳过已翻译页面

## 结论

- 韩中翻译链路完整可用，端到端实测 10 页平均 6.8s/页，吞吐量 8.8 页/分钟
- 性能与日中翻译持平（同为 14B 参数模型）
- 关键优化：禁用 Qwen3 thinking 模式，API 调用提速 10x
- 教训：Ollama 的 OpenAI 兼容端点不完全兼容，模型特定参数（如 `think`）需使用原生端点

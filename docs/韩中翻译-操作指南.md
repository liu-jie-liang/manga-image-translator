# 开发者技术指南 — 韩中翻译

> 韩→中翻译流水线的架构文档、开发环境和测试指南。用户使用文档见仓库 [README](../README.md)。日中翻译的开发者指南见 [开发者技术指南](./日中翻译-操作指南.md)。

## 1. 与日中流水线的技术差异

韩中翻译是一个独立的流水线，与日中翻译共享底层核心引擎（层 B/C）和 `batch_common.py` 工具库，但翻译层（层 A）完全不同：

```
共用  batch_common.py               ← 扫描、排序、进度读写
      ├── batch.py                   ← 日中批量编排
      └── batch_ko.py               ← 韩中批量编排（本流水线）

共用  层 B: manga_translator.py     ← 检测→OCR→擦除→渲染
共用  层 C: detection / OCR / inpainting

独有  translators/qwen3_kozh.py    ← Qwen3 韩中翻译器（本流水线核心）
```

| 维度 | 日中翻译 | 韩中翻译 |
|------|---------|---------|
| 入口 | `python -m manga_translator.batch` | `python -m manga_translator.batch_ko` |
| 翻译模型 | Sakura 14B / Galtransl 14B | Qwen3 14B |
| 翻译协议 | GGUF (llama-cpp-python) 或 Ollama HTTP | 仅 Ollama 原生 `/api/chat` |
| API 端点 | `/v1/chat/completions`（OpenAI 兼容） | `/api/chat`（Ollama 原生，非 OpenAI） |
| 降级链 | GGUF→Ollama→报错 | 无降级，Ollama 不可达直接报错 |
| 滑动窗口 | 已集成 | 未集成 |
| 质量评估 | LLM-as-Judge | 未集成 |
| 启动脚本 | 5 个日中脚本 | 1 个 `批量韩中翻译` |

## 2. 为什么用 Ollama 原生端点而非 OpenAI 兼容端点？

这是整个韩中流水线最关键的架构决策（[Iteration 7](./韩中翻译-迭代报告.md#iteration-7—qwen3-原生端点加速-2026-06-18)）。

**初始方案**：使用 Ollama 的 `/v1/chat/completions`（OpenAI 兼容端点），与日中翻译的方式A相同的 HTTP 调用模式。

**发现问题**：Qwen3 是推理模型，默认开启思考链（thinking tokens）。`/v1/chat/completions` 端点不支持 `enable_thinking` 参数，导致每次请求都生成大量不可控的思考 token。实测速度降至 **2.1 页/分钟**。

**最终方案**：切换到 Ollama 原生 `/api/chat` 端点，通过请求体中的 `"options": {"think": false}` 关闭思考：

```python
# qwen3_kozh.py — 原生 /api/chat 请求体
payload = {
    "model": model_name,
    "messages": messages,
    "stream": False,
    "options": {
        "temperature": translation_params.temperature,
        "top_p": translation_params.top_p,
        "think": False,  # 关键：禁用思考链
    }
}
```

**效果**：速度从 2.1 页/分钟恢复至 **8.8 页/分钟**（4.2 倍提升），与日中翻译的 GGUF 方案持平。

## 3. 翻译器实现 (`qwen3_kozh.py`)

```
class Qwen3KozhTranslator(CustomOpenAITranslator)
  ├── 继承 CustomOpenAITranslator 的基础设施（配置、语言映射）
  ├── __init__() 中从 CUSTOM_OPENAI_API_BASE 推导 Ollama 原生 API 地址
  ├── 覆写 _request_translation() → 用 aiohttp 直连 /api/chat
  │     ├── 构建 Ollama 原生请求体（含 think:false）
  │     ├── 设置 keep_alive='24h' 防止模型请求间被卸载
  │     └── 直接解析 /api/chat 响应格式
  ├── 覆写 chat_system_template → 韩中专用系统提示
  └── 覆写 get_chat_sample() → 韩中翻译示例
```

关键实现 —— `_request_translation()` 完全覆写父类，使用 `aiohttp` 直连 Ollama 原生 `/api/chat` 端点：

```python
async def _request_translation(self, to_lang: str, prompt: str) -> str:
    payload = {
        'model': self.model,
        'messages': messages,
        'stream': False,
        'keep_alive': '24h',   # 防止模型在请求间被卸载
        'think': False,         # 禁用思考模式
        'options': {
            'temperature': self.temperature,
            'top_p': self.top_p,
            'num_predict': self._MAX_TOKENS // 2,
        },
    }
    async with aiohttp.ClientSession() as session:
        async with session.post(self._ollama_chat_url, json=payload) as resp:
            data = await resp.json()
    return data['message']['content']
```

### 系统提示与翻译参数

翻译器使用韩文系统提示指定翻译方向和文体要求：韩文→简体中文、网络漫画（webtoon）风格、保留语气和格式、R18 内容直接翻译不审查。

推理参数：`temperature=0.3`、`top_p=0.3`、`num_predict=1024`（`_MAX_TOKENS // 2`），外加 `think=false` 硬编码在请求体中。超时 120s，无 HTTP 层重试（`_TIMEOUT_RETRY_ATTEMPTS=0`）。

## 4. 批量编排 (`batch_ko.py`)

与 `batch.py` 共享 `batch_common.py` 的扫描/排序/进度工具，差异仅在翻译器选择：

```
batch_ko.py
  ├── _probe_ollama()              # 探测 Ollama 是否可达（3s 超时）
  ├── _load_models()               # 加载 detection/OCR/inpainting（不含翻译模型）
  ├── _translate_directory()       # 调用 batch_common 工具逐目录翻译
  ├── batch_translate()            # 入口：清除进度→加载模型→遍历翻译→卸载
  └── main()                       # 交互询问路径和 retrans（支持 RETRANS 环境变量）
```

与 `batch.py` 的关键差异：

| | batch.py | batch_ko.py |
|------|---------|------------|
| 翻译器选择 | `_detect_translator_mode()` 探测 GGUF→Ollama 降级链 | 固定 `qwen3_kozh`，无探测 |
| 模型加载 | `_load_models()` 加载检测/OCR/擦除 + 本地 GGUF | `_load_models()` 加载检测/OCR/擦除（翻译由 Ollama 服务端管理） |
| 输入方式 | 交互式输入目录路径 | 交互式输入目录路径 |
| 滑动窗口 | 已集成 | 未集成 |
| 翻译重试 | `attempts=1`（无重试） | `attempts=1`（无重试） |

## 5. 开发环境搭建

### 前置条件

韩中翻译的依赖比日中翻译更轻量——不需要 `llama-cpp-python`（无 GGUF 本地加载）：

```bash
# 核心依赖（与日中相同）
pip install -r requirements.txt

# 韩中翻译不需要 requirements-gguf.txt
# 但需要 Ollama 服务
```

### 安装 Ollama 并拉取模型

```bash
# 安装 Ollama (macOS/Linux/Windows 均支持)
# https://ollama.com

# 拉取 Qwen3 14B
ollama pull qwen3:14b-q4_k_m

# 验证
curl http://localhost:11434/api/tags | grep qwen3
```

### 环境变量

```bash
# bash/zsh — 开发环境最小集
export CUSTOM_OPENAI_API_BASE="http://localhost:11434/v1"
export CUSTOM_OPENAI_MODEL="qwen3:14b-q4_k_m"
export CUSTOM_OPENAI_API_KEY="ollama"
```

```cmd
:: CMD
set CUSTOM_OPENAI_API_BASE=http://localhost:11434/v1
set CUSTOM_OPENAI_MODEL=qwen3:14b-q4_k_m
set CUSTOM_OPENAI_API_KEY=ollama
```

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `CUSTOM_OPENAI_API_BASE` | Ollama 服务地址（含 `/v1`） | `http://localhost:11434/v1` |
| `CUSTOM_OPENAI_MODEL` | Ollama 模型名 | `qwen3:14b-q4_k_m` |
| `CUSTOM_OPENAI_API_KEY` | Ollama API Key | `ollama` |
| `RETRANS` | `true`=全量重翻, `false`=续传 | 交互式选择 |

> `CUSTOM_OPENAI_API_BASE` 建议设为 Ollama 的 `/v1` 端点地址（如 `http://localhost:11434/v1`）。`qwen3_kozh.py` 的 `__init__()` 会自动去掉 `/v1` 并推导出 `/api/chat` 的原生端点地址。不带 `/v1` 的地址（如 `http://localhost:11434`）同样兼容。

## 6. 开发工作流

### 运行入口

```bash
# 命令行启动（交互式输入目录路径）
python -m manga_translator.batch_ko

# 或通过启动脚本
bash start-scripts/linux/批量韩中翻译.sh
```

### 启动脚本

韩中翻译仅有一个脚本变体（无 GGUF/Galtransl/全量/续传等变体）：

| 平台 | 路径 |
|------|------|
| macOS | `start-scripts/macos/批量韩中翻译.command` |
| Linux | `start-scripts/linux/批量韩中翻译.sh` |
| Windows | `start-scripts/windows/批量韩中翻译.bat` |

脚本等价逻辑：

1. `cd` 到项目根目录
2. 激活 Python 环境（conda → venv → 系统 Python）
3. 设置 `CUSTOM_OPENAI_API_BASE`、`CUSTOM_OPENAI_MODEL`、`CUSTOM_OPENAI_API_KEY`
4. 运行 `python -m manga_translator.batch_ko`

### 预期效果

| 指标 | 值 |
|------|-----|
| 翻译模型 | Qwen3 14B Q4_K_M |
| 端到端速度 | ~6.8 秒/页（8.8 页/分钟） |
| 翻译质量 | 韩→中翻译流畅，webtoon 风格适配 |
| 输出命名 | `[原目录名] 汉化`（同级目录） |
| 续传 | 支持（`.translate_progress.json`） |

## 8. 故障排查

### Ollama 不可达

```bash
# 确认服务状态
curl http://localhost:11434/api/tags

# 未启动则启动
ollama serve
```

### 模型未找到

```bash
ollama pull qwen3:14b-q4_k_m
curl http://localhost:11434/api/tags | grep qwen3
```

### 翻译速度异常慢

确认 `_ollama_chat_url` 指向的是 `/api/chat` 而非 `/v1/chat/completions`。如果请求走了 OpenAI 兼容端点，检查 `CUSTOM_OPENAI_API_BASE` 的值是否正确（`qwen3_kozh.py` 会自动从 `/v1` 推导出 `/api/chat`）。

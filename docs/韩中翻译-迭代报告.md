### Iteration 7 — 翻译性能修复（2026-06-18）

| 项目 | 内容 |
|------|------|
| 问题 | 韩中翻译极慢，2.1 页/分钟，每页约 28.7s |
| 根因 | `/v1/chat/completions` 端点不支持 `enable_thinking` 参数，Qwen3 模型仍在生成大量思考 token |
| 修复 | 切换为 Ollama 原生 `/api/chat` 端点，使用 `think: false` 参数 |
| 效果 | 8.8 页/分钟，每页约 6.8s，**提速 4.2 倍** |

**问题定位过程**：

1. 用户反馈"韩中翻译好慢，一分钟 2-3 页"
2. 检查代码发现 `extra_body={'enable_thinking': False}` 已设置
3. 首次怀疑：`batch_ko.py` 使用了 `translator: 'custom_openai'` 而非 `'qwen3_kozh'`，导致泛型类没有 `extra_body` 属性
4. 修复后注册专用翻译器 `qwen3_kozh`，确认 `extra_body` 已传入 API 请求
5. 但实测速度仍慢（28.7s/页），说明 `enable_thinking` 参数本身未被 Ollama 处理
6. 用 curl 直接测试 Ollama `/api/chat` 端点，发现 `think: false` 能将响应从 10s 降到 0.28s
7. 根因确认：**`/v1/chat/completions`（OpenAI 兼容端点）不支持 `think` 或 `enable_thinking` 参数**，只有 Ollama 原生 `/api/chat` 端点支持
8. 修改 `Qwen3KoZhTranslator._request_translation` 使用 `aiohttp` 直连 `/api/chat`，传 `"think": false`

## 架构决策

### 决策 1：抽取公共模块 batch_common.py

**问题**：`batch.py` 和 `batch_ko.py` 有大量重复逻辑（目录排序、图片扫描、进度管理）。

**决策**：抽取公共模块 `batch_common.py`，`batch.py` 和 `batch_ko.py` 均从此导入。

**收益**：
- 减少代码重复，公共逻辑只维护一份
- 两者共用相同的目录排序规则和进度文件格式
- 日中翻译功能不受影响（153 个单元测试全部通过，验证无回归）

### 决策 2：选择 Qwen3 14B 作为韩中翻译模型

**问题**：需要选择一个适合韩中漫画翻译的模型。

**候选方案**：
| 方案 | 翻译器 | 优点 | 缺点 |
|------|--------|------|------|
| A | Papago API | 免费、原生韩中 | 需要 API 调用、网络依赖 |
| B | DeepSeek API | LLM 翻译 | 付费 |
| C | Qwen3 14B (Ollama) | 本地部署、开源免费、已在环境中 | 需要 Ollama 服务 |

**决策**：选择方案 C（Qwen3 14B），理由：
- 用户环境已部署 Ollama + `qwen3:14b-q4_k_m`
- 无需额外 API Key 或付费
- 与日中翻译的 Sakura 模型部署方式一致，降低维护成本
- Qwen3 支持多语言，韩中翻译质量可接受

### 决策 3：无降级链

**问题**：韩中翻译是否需要像日中翻译一样的降级链？

**决策**：不设降级链，Ollama 不可达时直接报错退出。

**理由**：
- 用户明确要求"没有降级，报错就退出"
- 韩中翻译暂无可靠的第二翻译方案
- 简化架构，减少维护负担

### 决策 4：禁用 thinking 模式（Iteration 7 修复）

**问题**：Qwen3 模型默认启用思考模式，每次翻译请求会生成 200-300 个思考 token（约 10-12s），仅输出 2-3 个实际翻译 token。

**尝试 1（失败）**：通过 `extra_body = {'enable_thinking': False}` 禁用思考模式。

**失败原因**：`Qwen3KoZhTranslator` 继承 `CustomOpenAiTranslator`，使用 `openai.AsyncOpenAI` 客户端调用 `/v1/chat/completions` 端点。Ollama 的 OpenAI 兼容端点**不支持 `enable_thinking` 或 `think` 参数**，会静默忽略。

**尝试 2（部分修复）**：注册专用翻译器 `qwen3_kozh` 确保 `extra_body` 被传入 API 请求。

**失败原因**：虽然 `extra_body` 被正确传入，但 `/v1/chat/completions` 端点仍不处理该参数。问题不在参数传递，而在端点本身。

**最终修复（成功）**：

1. 使用 curl 直接测试 Ollama 原生 `/api/chat` 和 `/v1/chat/completions` 端点
2. 发现原生 `/api/chat` 端点支持 `"think": false` 参数，10s → 0.28s
3. 修改 `_request_translation` 使用 `aiohttp` 直连 `/api/chat`，传 `"think": false`
4. 移除不再需要的 `extra_body` 机制

**效果**：
- API 调用：15-27s → 0.6-2.9s（~10x 提速）
- 单页翻译：28.7s → 6.8s（4.2x 提速）
- 页/分钟：2.1 → 8.8

**教训**：
- Ollama 的 OpenAI 兼容端点并非完全兼容，模型特定参数（如 `think`）只有原生端点支持
- 使用 LLM 翻译时，务必用 curl 直接测试 API 端点确认参数生效
- 代码中设置参数 ≠ 参数被 API 端点处理

## 已知限制

1. **Ollama 依赖**：韩中翻译完全依赖 Ollama 服务，不可用时无法翻译
2. **无滑动窗口**：当前韩中翻译未集成滑动窗口上下文翻译策略
3. **无质量评估**：没有 LLM-as-Judge 质量评分模块
4. **单模型**：只有 Qwen3 一个翻译器，没有备选方案
